# 蓝桥杯纠错命令 (/lanqiao-fix)

双模式自适应纠错系统：自动检测 fix-report.md 切换模式。
- **经验模式（首次）**：经验库智能筛选 → 核心清单兜底 → 开放分析 → 实战经验自动积累
- **深度模式（二次）**：已有报告解析 → 错误定位 → 5维度区域深排 → 修复连带验证 → 经验反哺

## 核心规则

1. **理解驱动** — 先读题目，理解代码应该做什么，再判断代码哪里做错了
2. **全量读取** — 用户代码全部读取（嵌入式读 APP/ + main.c 用户区；算法读全部源码）
3. **三方交叉** — 题目要求 + 代码实现 + 错误现象，三方对照验证后才下结论
4. **最小修改** — 只改必要的行，不做无关重构
5. **备份原始** — 修复前在报告中记录原始代码（仅备份被修改的文件）
6. **保留历史** — 若 `fix-report.md` 已存在，先读取保留已有修复记录，新修复以追加方式写入，编号延续
7. **经验驱动** — 优先使用经验库中高命中率的模式进行检查，提高纠错效率
8. **双模式自适应** — 自动检测 `fix-report.md` 存在性切换模式：不存在→经验模式（首次纠错），已存在→深度模式（残余排查）

## 参数

- `<文件夹路径>` (必需): 包含题目文件、截图和源码的文件夹路径

## 执行流程

### Step 1: 文件发现 + 项目类型识别 + 加载经验库 + 模式判断

用 Glob 扫描 `$ARGUMENTS` 路径，**只列出文件清单并分类，不读取内容**：

```
Glob(pattern="**/*.{png,jpg,jpeg,bmp,gif}", path=$ARGUMENTS)     → 截图列表
Glob(pattern="**/*.{c,cpp,h,hpp}", path=$ARGUMENTS)               → 源码列表
Glob(pattern="**/*.{ppt,pptx}", path=$ARGUMENTS)                  → PPT 列表
Glob(pattern="**/*.{pdf}", path=$ARGUMENTS)                        → PDF 列表
Glob(pattern="**/*.{doc,docx}", path=$ARGUMENTS)                   → Word 列表
Glob(pattern="**/*.{md}", path=$ARGUMENTS)                         → MD 列表（排除 fix-report.md）
Glob(pattern="**/*.{ioc}", path=$ARGUMENTS)                        → IOC 列表
```

**项目类型自动识别**：
- 存在 `*.ioc` 或 `Core/Src/main.c` 或 `APP/` 目录 → **嵌入式模式** (`embedded`)
- 否则 → **算法模式** (`algorithm`)

校验：
- 无源码文件 → 输出错误并终止: "未找到 C/C++ 源码文件，请确认文件夹路径正确"
- 无截图且无 PPT/PDF/MD → 警告但继续: "未找到题目文件和截图，将仅做源码静态分析"

#### 模式判断（核心分支）

检查 `$ARGUMENTS/fix-report.md` 是否已存在：

- **不存在** → **经验模式**：走 Step 2 → Step 3 → Step 4 → Step 5
- **已存在** → **深度模式**：走 Step 2 → Step 2D → Step 3D → Step 4（追加） → Step 5

深度模式触发时，立即读取 `fix-report.md` 内容供后续 Step 2D 解析。

#### 加载经验库

读取 `~/.claude/lanqiao-fix-experience.json`：
- **文件存在且格式正确** → 加载成功，输出概况
- **文件不存在** → 警告"经验库未找到，将仅使用核心清单"，继续执行
- **格式损坏** → 警告"经验库格式异常，将仅使用核心清单"，继续执行

输出：
```
文件清单: 源码 X 个, 截图 Y 张, PPT/PDF Z 个
项目类型: [嵌入式/算法]
模式: [经验模式（首次纠错）/ 深度模式（已有N处修复）]
经验库: 已加载 N 条模式（历史 M 次修复，S 次会话）
```

### Step 2: 读取题目要求 + 提取上下文标签

**无条件读取所有题目文件**，按以下顺序尝试：

1. **PPT/PDF 文件** — 使用 PPT 解析策略（见下方）或 Read 直读 PDF
2. **MD 文件** — Read 直接读取（排除 fix-report.md）
3. **Word 文件** — 尝试 python-docx 解析，降级为 Read 直读
4. **截图文件** — Read 读取所有截图，进行**智能分类**

#### 截图智能分类

对每张截图，根据内容分类为：
- **题目截图** — 含功能描述、评分标准、原理图、引脚定义、硬件连接图、操作说明
- **错误截图** — 含编译输出、运行结果、调试信息、终端输出、diff 对比

分类依据：
- 有明确的功能要求描述（如"按下B1按键..."、"LCD显示..."） → 题目截图
- 有评分标准或分值 → 题目截图
- 有电路图/原理图/引脚图 → 题目截图
- 有编译器输出（error/warning）、程序运行界面 → 错误截图
- 有 expected vs actual 对比 → 错误截图
- 无法判断 → 同时归入两类

#### 输出结构化题目理解摘要

从题目文件中提取并整理：
- **功能要求清单**：逐条列出每个功能点及其评分（如有）
- **外设要求**：使用了哪些外设（LCD、LED、按键、ADC、UART、PWM、I2C 等）
- **输入输出规格**：数据格式、显示格式、精度要求
- **约束条件**：时序要求、边界值、特殊规则

#### 提取上下文标签

从题目功能要求中自动提取标签，用于经验库智能筛选：

**标签提取规则**：
| 类别 | 提取规则 |
|------|----------|
| 外设标签 | 提到 LCD/LED/UART/PWM/ADC/DAC/I2C/SPI/按键/定时器/DMA/EEPROM → 加对应标签 |
| 功能标签 | 提到 闪烁/计时/累积统计/数据存储/格式显示/输入捕获/命令处理/状态机 → 加对应标签 |
| 错误类型 | 代码中出现可疑模式时标记：类型溢出/逻辑错误/格式错误/初始化缺失/数据残留/边界错误 |

输出：`本题标签: [LCD, UART, PWM, 按键, 状态机, 命令处理]`

#### PPT 解析策略（两级降级）

**策略 1 — python-pptx 解析（优先）：**

```bash
python -c "import pptx" 2>/dev/null || pip install python-pptx
```

```bash
python -c "
from pptx import Presentation
import sys
prs = Presentation(sys.argv[1])
for i, slide in enumerate(prs.slides, 1):
    print(f'--- 第{i}页 ---')
    for shape in slide.shapes:
        if shape.has_text_frame:
            for para in shape.text_frame.paragraphs:
                text = para.text.strip()
                if text:
                    print(text)
        if shape.has_table:
            table = shape.table
            for row in table.rows:
                cells = [cell.text.strip() for cell in row.cells]
                print(' | '.join(cells))
" "<PPT文件路径>"
```

**策略 2 — 降级直读：**

如果 python-pptx 解析失败，用 Read 工具直接读取 PPT 文件，并在报告中注明"PPT 解析受限，以下信息可能不完整"。

#### 模式分支标记

至此 Step 2 完成。根据 Step 1 的模式判断：
- **经验模式** → 继续 Step 3（经验驱动分析）
- **深度模式** → 跳转 Step 2D（已有报告解析）

### Step 2D: 已有报告解析 + 错误定位（仅深度模式）

> 本步骤仅在**深度模式**下执行。

#### 2D.1 解析已有 fix-report.md

从 Step 1 已读取的 `fix-report.md` 中提取：

1. **已修复列表**：提取每个修复的编号、文件、行号、错误类型、来源
2. **最大修复编号 N**：后续新修复从 N+1 开始编号
3. **已覆盖的经验模式 ID**：记录已命中的 EXP-xxx / SEED-xxx ID，避免重复检查
4. **功能覆盖状态**：提取每个功能点的状态（✅/⚠️/❌）
5. **已分析的截图列表**：记录已在报告中出现的截图文件名

输出：
```
已有报告解析:
- 已修复: N 处（编号 1~N）
- 已覆盖经验模式: [EXP-001, SEED-012, ...]
- 功能覆盖: ✅X个 ⚠️Y个 ❌Z个
- 已分析截图: [a.png, b.png, ...]
```

#### 2D.2 新截图错误定位

**识别新增截图**：对比当前截图列表与已有报告记录的截图，找出新增的截图。

**从新错误截图提取现象并映射嫌疑区域**：

| 错误现象 | 嫌疑区域 |
|----------|---------|
| LCD显示异常（内容错误/格式不对/不刷新） | LCD显示函数 + 数据源变量 + sprintf格式化 + 刷新逻辑 |
| LED状态不对（不亮/常亮/闪烁异常） | LED控制函数 + 状态机逻辑 + 定时器回调 + GPIO配置 |
| 按键无响应/误触发 | 按键扫描函数 + 消抖逻辑 + 事件处理 + 中断配置 |
| UART收发异常（乱码/丢包/无响应） | UART中断/DMA配置 + 数据解析 + 缓冲区管理 + 波特率 |
| PWM输出不对（频率/占空比错误） | 定时器配置 + 占空比计算 + ARR/PSC/CCR寄存器 |
| 数值计算错误（结果偏差/溢出） | 变量类型 + 运算顺序 + 整数除法 + 浮点精度 + 溢出 |
| 功能不触发（条件满足但无动作） | 任务调度表 + 回调注册 + 条件判断逻辑 + 标志位管理 |
| ADC/DAC读数异常 | ADC通道配置 + 采样时间 + 参考电压 + 转换公式 |
| 定时器行为异常 | 定时器初始化 + 预分频 + 中断使能 + 回调函数 |
| EEPROM读写异常 | I2C配置 + 地址计算 + 页写入边界 + 延时 |

**无新截图时的退化策略**：

如果没有新增截图，基于已有报告中的缺陷项（⚠️/❌ 功能项）定位嫌疑区域：
- ⚠️ 项：对应功能的实现代码模块
- ❌ 项：对应功能应在的代码位置（可能是缺失实现）
- 如果所有功能都是 ✅ 且无新截图，执行**全模块逐项排查**

输出：
```
嫌疑区域定位:
- 区域1: [模块名] — 来源: [新截图xxx.png / ⚠️功能项X / ❌功能项Y]
  现象: [具体错误现象]
  嫌疑文件: [file1.c, file2.c]
- 区域2: ...
```

### Step 3: 全量读取用户代码 + 经验驱动分析（经验模式）

> 本步骤在**经验模式**下执行。深度模式跳转 Step 3D。

#### 源码读取策略（按项目类型）

**嵌入式模式**：
- **必读**：
  - `APP/`（或 `app/`）下全部 `.c` 和 `.h` 文件
  - `Core/Src/main.c` 和 `Core/Inc/main.h` 的 `USER CODE` 区域（`/* USER CODE BEGIN */` 到 `/* USER CODE END */` 之间的内容）
- **不读**：
  - `Drivers/` 目录（HAL/LL 库，不会被改动）
  - `CMSIS/` 目录
  - `*.s` 启动文件
  - `Middlewares/` 目录
- **可选**（问题涉及时）：
  - 外设初始化文件（`gpio.c`、`tim.c`、`adc.c` 等，在 CubeMX 生成的 USER CODE 区可能有配置）
  - `stm32xxx_it.c` 中断处理文件的 USER CODE 区

**算法模式**：
- 读取所有 `.c`/`.cpp`/`.h`/`.hpp` 源码文件

#### 读取错误截图

读取 Step 2 分类为"错误截图"的图片，提取错误信息：
- 编译错误消息（error/warning 文本）
- 运行时异常（HardFault、段错误等）
- 输出对比（期望 vs 实际）
- 调试器信息

#### 三方交叉分析（经验驱动，三阶段）

##### 阶段 A：经验库智能筛选

1. **筛选**：从经验库中筛选 `tags` 与当前题目标签有交集的模式，以及 `category == "universal"` 的通用模式
2. **排序**：按 `hit_count` 降序（高频命中优先检查）
3. **逐条检查**：对每个匹配的经验模式，检查代码中是否存在其 `signature` 描述的错误特征
4. **记录命中**：命中的模式记入 `hit_list`（后续更新 hit_count）

输出：`经验库筛选: 从 N 条中匹配 M 条（按题目标签 [LCD, UART, ...] 筛选）`

##### 阶段 B：核心清单兜底

检查以下精简核心清单（P0 致命级，任何题目都必须检查）。**跳过阶段 A 已覆盖的项目**。

**嵌入式核心（6项）**：
- [ ] switch-case 是否缺少 break（case 穿透）
- [ ] 变量类型是否会溢出（uint8_t 存 >255 的值，应为 uint16_t/uint32_t）
- [ ] 整数除法顺序是否正确（`(float)(a/b)` 应为 `(float)a/(float)b`）
- [ ] sprintf 是否有缓冲区溢出风险（无长度限制）
- [ ] 函数/回调/任务已定义但未注册到调度器任务列表或回调表
- [ ] sscanf 格式符宽度与目标变量类型不匹配（`%d` 写4字节到 uint16_t → 栈溢出）

**算法核心（4项）**：
- [ ] 数组越界（数组大小不够、下标从 0/1 混淆）
- [ ] 整数溢出（int 应该用 long long）
- [ ] 格式符错误（%d 与 %lld 不匹配）
- [ ] 边界条件遗漏（n=0、n=1 等特殊情况）

##### 阶段 C：开放分析

基于三方交叉分析（题目↔代码↔错误），发现不在经验库和核心清单中的新问题：

1. **题目↔代码**：题目要求的每个功能，代码是否实现？实现是否正确？
2. **代码↔错误**：错误现象指向代码的哪个位置？根因是什么？
3. **题目↔错误**：错误是否因为误解题意导致？

新发现的问题标记为"新发现"，记入 `new_discoveries` 列表。

列出所有发现的问题，按严重程度排序（P0 → P3）。

### Step 3D: 5维度区域深度排查（仅深度模式）

> 本步骤仅在**深度模式**下执行，替代 Step 3 的经验驱动分析。

#### 源码读取

同 Step 3 的源码读取策略，全量读取用户代码。

#### 5维度深度排查

对 Step 2D 定位的每个嫌疑区域，执行以下 5 个维度的深度分析：

##### 维度1：数据流追踪

对嫌疑区域涉及的关键变量，追踪完整的数据流路径：

- **定义** → **初始化** → **赋值** → **传递** → **使用** → **输出**
- 检查点：
  - [ ] 类型匹配：赋值两侧类型是否兼容？是否有隐式截断？
  - [ ] 溢出风险：运算结果是否超出变量类型范围？
  - [ ] 精度损失：整数除法、浮点截断、类型转换是否丢精度？
  - [ ] 未初始化：变量是否在所有路径上都有初始值？
  - [ ] 数据残留：全局变量/缓冲区是否在切换场景时正确清零？
  - [ ] 格式化匹配：sprintf/sscanf 格式符是否与变量类型一致？

##### 维度2：状态机验证

如果嫌疑区域包含状态机逻辑（显式 enum/switch 或隐式 if-else 链）：

- **枚举所有状态**：列出所有可能的状态值
- **绘制状态转移矩阵**：

| 当前状态 \ 事件 | 事件A | 事件B | 事件C |
|----------------|-------|-------|-------|
| 状态1 | → 状态2 | 保持 | → 状态3 |
| 状态2 | ... | ... | ... |

- 检查点：
  - [ ] 不可达状态：是否存在永远无法到达的状态？
  - [ ] 死锁状态：是否存在无法退出的状态？
  - [ ] 非法状态：default/else 分支是否处理了意外状态值？
  - [ ] 转移条件互斥性：同一状态的多个转移条件是否有交叉？
  - [ ] 状态初始化：初始状态是否正确设置？

##### 维度3：边界条件

对嫌疑区域的所有数据操作检查边界情况：

- 检查点：
  - [ ] 循环边界：for/while 的起止条件是否正确？off-by-one？
  - [ ] 数组越界：下标是否可能超出数组大小？
  - [ ] 除零保护：除数是否可能为零？
  - [ ] 空值/空串：字符串操作是否处理了空串？指针是否可能为 NULL？
  - [ ] 极端值：最大值/最小值输入下行为是否正确？
  - [ ] 缓冲区边界：缓冲区写入是否有长度检查？环形缓冲区是否正确回绕？
  - [ ] 计数器边界：计数器递增/递减是否有上下限保护？

##### 维度4：时序逻辑

对嵌入式系统特有的时序问题进行检查：

- 检查点：
  - [ ] 中断优先级：嵌套中断的优先级配置是否正确？
  - [ ] volatile 标记：中断/主循环共享变量是否声明为 volatile？
  - [ ] 原子操作：多字节变量在中断中的读写是否需要临界区保护？
  - [ ] 定时器周期：定时器中断频率是否与预期一致？ARR/PSC 计算是否正确？
  - [ ] 回调上下文：回调函数中是否有耗时操作？是否在正确的上下文中执行？
  - [ ] 消抖逻辑：按键消抖时间是否足够？消抖状态机是否正确？
  - [ ] 启动顺序：外设初始化顺序是否正确？是否有先使用后初始化的情况？
  - [ ] DMA 同步：DMA 传输完成标志是否正确检查？缓冲区是否在传输期间被修改？

##### 维度5：模块间交互

检查不同模块/文件之间的协作正确性：

- 检查点：
  - [ ] 全局变量一致性：同一全局变量在不同文件中的声明/定义是否一致（类型、extern）？
  - [ ] 函数参数范围：调用方传入的参数值是否在被调用方的预期范围内？
  - [ ] 头文件声明匹配：.h 声明与 .c 定义是否一致（参数类型、返回值）？
  - [ ] 回调注册完整性：所有需要注册的回调是否都已注册？注册参数是否正确？
  - [ ] 初始化依赖：模块 A 依赖模块 B 的初始化，B 是否先于 A 初始化？
  - [ ] 数据一致性：多个模块共享的数据结构，修改方和读取方是否使用相同的字段含义？
  - [ ] 接口契约：函数返回值的含义是否被调用方正确理解（0=成功 vs 0=失败）？

#### 修复连带验证

5维度排查完成后，额外执行**连带验证**——检查经验模式（第1轮）的已有修复是否引入新问题：

1. **逐条审查已有修复**：
   - 修复后的代码是否引入新的类型不匹配？
   - 修复是否破坏了其他功能的正确性？
   - 修复是否遗漏了相关联的代码位置（如多处使用同一变量，只改了一处）？

2. **交叉影响检查**：
   - 修复 A 是否与修复 B 产生冲突？
   - 多处修复叠加后的整体行为是否正确？

输出：
```
5维度排查结果:
- 区域1 [模块名]:
  - 维度1(数据流): ✅通过 / ⚠️发现问题X
  - 维度2(状态机): ✅通过 / N/A（无状态机）
  - 维度3(边界): ✅通过 / ⚠️发现问题Y
  - 维度4(时序): ✅通过 / ⚠️发现问题Z
  - 维度5(交互): ✅通过 / ⚠️发现问题W
- 区域2: ...
- 连带验证: ✅无连带问题 / ⚠️发现N处连带问题
新发现问题: K 处（按严重程度 P0→P3 排序）
```

### Step 4: 修复 + 生成报告

#### 代码修复

按优先级逐一修复：**P0 致命 → P1 高危 → P2 中危 → P3 低危**

对每处修复：
1. 记录原始代码（文件、行号、原始内容）
2. 使用 Edit 工具执行最小修改
3. 记录修改理由，**引用题目要求**说明为什么这样改

```
Edit(file_path="<源码路径>", old_string="<原始代码>", new_string="<修复代码>")
```

修复原则：
- 只改必要的行，不改风格、不加无关注释
- 如果修复不确定，在报告中标注"建议修复，需验证"
- 如果存在多种修复方案，选择最简单的
- **深度模式下**：修复编号从已有报告最大编号 N+1 开始延续

#### 功能覆盖核查

修复完成后，对照题目要求清单逐条核查：
- ✅ 已实现且正确
- ⚠️ 已实现但有问题（已修复/建议修复）
- ❌ 未实现（提示用户需补充）

##### 经验模式：新建报告

收到用户确认（"保存"）后：

1. **更新已命中模式**：
   - `hit_count += 1`
   - `last_hit` 更新为今天日期

2. **写入新发现模式**：
   - 分配 `EXP-xxx` ID（从当前最大 EXP 编号 +1 开始）
   - `confidence: "confirmed"`

   - **深度模式额外字段**：

   - **经验模式字段**：
     - `discovery_method: "experience_match" | "checklist" | "open_analysis"` — 按来源标记

   - `stats.total_sessions += 1`
   - `stats.total_fixes += N`（本次修复数）
   - `stats.total_patterns` 更新为当前总数
   - `stats.last_updated` 更新为今天
   - **新增统计**：`stats.deep_mode_sessions` 记录深度模式使用次数

   ```

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 路径不存在 | 报错终止: "路径不存在: $ARGUMENTS" |
| 无 C/C++ 源码 | 报错终止: "未找到 C/C++ 源码文件" |
| PPT 解析失败 | 降级到 Read 直接读取，报告中注明 |
| 无题目文件 | 警告继续: "未找到题目文件，将仅基于代码和错误截图分析" |
| 无截图 | 正常继续，仅做题目↔代码的双向分析 |
| 截图无法识别 | 记录"截图内容无法识别"到报告，继续处理其他文件 |
| Write 报告失败 | 将报告内容直接输出到对话中 |
| 经验库文件不存在 | 警告并仅用核心清单，继续执行 |
| 经验库格式损坏 | 警告并仅用核心清单，继续执行 |
| 经验库写入失败 | 将更新内容输出到对话中，提示用户手动保存 |
| fix-report.md 格式损坏/为空 | 退化为经验模式，输出警告 |
| 无新截图且无缺陷功能项 | 执行全模块逐项排查（5维度覆盖所有源码模块） |
| 嫌疑区域定位失败 | 退化为全模块排查，输出警告 |
