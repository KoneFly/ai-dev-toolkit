# lanqiao-fix — 蓝桥杯竞赛自进化纠错系统

> 练得越多、改得越快 — 像学生的**错题本**一样，自动积累、智能调用、越用越准。

## 功能概述

`/lanqiao-fix` 是一个面向蓝桥杯嵌入式/算法竞赛的 Claude Code 纠错技能。给定一道竞赛题的文件夹（含题目 PDF/PPT、源代码、错误截图），它能自动分析代码中的 bug 并修复。

**核心特性**：

- **双模式自适应** — 自动检测 fix-report.md 切换：首次→经验模式，二次→深度模式（5维度排查）
- **经验库驱动** — 74 条经验模式按标签智能筛选，高命中率模式优先检查
- **自进化积累** — 每次使用后自动学习新模式，深度模式发现的模式带 `discovery_method` 标记
- **三阶段分析** — 经验库筛选 → 核心清单兜底 → 开放分析，不遗漏不重复
- **5维度深排** — 数据流追踪 / 状态机验证 / 边界条件 / 时序逻辑 / 模块交互（深度模式专属）
- **全格式题目** — 支持 PDF/PPT/Word/Markdown/截图，自动分类题目截图与错误截图
- **最小修改** — 只改必要的行，修复前备份原始代码

## 快速开始

### 安装

```bash
# 1. 复制技能文件到 Claude Code 命令目录
cp lanqiao-fix.md ~/.claude/commands/

# 2. 复制经验库到 Claude Code 配置目录
cp lanqiao-fix-experience.json ~/.claude/
```

### 使用

在 Claude Code 中执行：

```
/lanqiao-fix D:/lanqiao/problem01
```

文件夹中应包含：
- 题目文件（PDF / PPT / MD / 截图）
- 源代码（`.c` / `.h` 文件）
- 错误截图（可选，用于辅助定位）

### 输出

- 自动修复代码中的 bug（使用 Edit 工具最小修改）
- 生成 `fix-report.md` 修复报告（含原始代码备份）
- 输出经验学习摘要（新模式需用户确认后存入）

### v1.0 — 静态清单（2026-03-10）

**定位**：最小可用版本，验证"题目+代码+截图→自动纠错"的基本流程。

| 维度 | 内容 |
|------|------|
| 分析方式 | 硬编码 30+ 条检查项（P0~P3），逐条遍历 |
| 工作流 | 4步：文件发现 → 读题 → 读码+分析 → 修复+报告 |
| 学习能力 | 无 — 每次都用同一套清单 |
| 报告格式 | 基础修复报告 |
| 技能文件 | ~200行 |

**局限**：
- 检查项硬编码，无法按题目特征筛选，大量无关检查浪费上下文
- 无法从实战中积累新模式，发现的新 bug 类型下次用不上
- 单一分析模式，首次和二次调用走同样流程

### v2.0 — 经验库驱动（2026-03-14）

**定位**：引入外部经验库 + 自进化机制，从"死记硬背"升级为"活学活用"。

| 维度 | v1.0 → v2.0 变化 |
|------|-------------------|
| 分析方式 | 硬编码清单 → **JSON 经验库按标签智能筛选** + 核心清单兜底 + 开放分析 |
| 工作流 | 4步 → **5步**（新增 Step 5 经验学习） |
| 学习能力 | 无 → **自进化**：命中模式 hit_count+1，新发现经用户确认后存入 |
| 经验库 | 无 → **74 条模式**（34 SEED + 40 EXP） |
| 报告格式 | 基础报告 → **含经验命中统计** |
| 技能文件 | ~200行 → **445行** |

**关键新增**：

```
                    ┌─────────────────────────────┐
                    │    lanqiao-fix-experience.json│
                    │    74 条模式 (SEED + EXP)     │
                    └──────────┬──────────────────┘
                               │ 按题目标签筛选
                               ▼
阶段A: 经验库匹配 ──▶ 阶段B: 核心清单兜底 ──▶ 阶段C: 开放分析
(高频优先)            (P0永远不跳过)           (三方交叉)
                               │
                               ▼
                    Step 5: 经验学习
                    ├─ 命中: hit_count +1
                    └─ 新发现: 用户确认 → EXP-041...
```

**数据来源**：

| 来源 | ID 范围 | 说明 |
|------|---------|------|
| 静态清单迁移 | SEED-001~034 | 从 v1.0 的 P1~P3 检查项迁移 |
| 15届国赛实战 | EXP-001~010 | 3.4题 STM32 运动控制系统，7轮26处修复 |
| 满分代码分析 | EXP-011~030 | 12~17届省赛/模拟题 10 个满分项目 + 17 张错误截图 |
| 修复笔记提取 | EXP-031~040 | 10 份修复笔记，共约 100 处修复记录 |

### v3.0 — 双模式自适应（2026-03-21）⬅ 当前版本

**定位**：解决"二次调用仍走经验匹配，残余复杂问题排查力度不足"的痛点。

| 维度 | v2.0 → v3.0 变化 |
|------|-------------------|
| 分析方式 | 单一模式 → **双模式自适应**（经验模式 + 深度模式） |
| 模式切换 | 无 → **自动检测 fix-report.md** 存在性切换 |
| 深度分析 | 无 → **5维度区域深排**（数据流/状态机/边界/时序/交互） |
| 连带验证 | 无 → **检查第1轮修复是否引入新问题** |
| 报告格式 | 覆盖式 → **追加式**（深度模式编号延续，不覆盖第1轮） |
| 经验反哺 | 单一来源 → **discovery_method + analysis_dimension** 标记 |
| 错误处理 | 10条 → **13条**（报告损坏退化、无截图全排查、定位失败退化） |
| 技能文件 | 445行 → **749行**（+304行） |

**关键新增**：

```
Step 1: 文件发现 + 项目识别 + 加载经验库 + 模式判断
         │
         ├─ fix-report.md 不存在 → 经验模式（首次纠错）
         │   ↓
         │   Step 2: 读取题目 + 提取上下文标签
         │   Step 3: 经验驱动三阶段分析（筛选→兜底→开放）
         │   Step 4: 修复 + 新建报告
         │
         └─ fix-report.md 已存在 → 深度模式（残余排查）
             ↓
             Step 2:  读取题目 + 提取上下文标签
             Step 2D: 已有报告解析 + 错误现象→嫌疑区域映射
             Step 3D: 5维度区域深排 + 修复连带验证
             Step 4:  追加式报告（编号延续，不覆盖第1轮）
         ↓
Step 5: 经验学习（命中 +1，新发现确认存入，深度模式带 discovery_method）
```

**v3.0 新增组件详解**：

#### Step 2D：已有报告解析 + 错误定位

| 步骤 | 作用 |
|------|------|
| 解析 fix-report.md | 提取已修复列表、最大编号、已覆盖经验ID、功能覆盖状态 |
| 新截图识别 | 对比已有报告记录，找出新增截图 |
| 错误现象→嫌疑区域映射 | 10类现象（LCD/LED/按键/UART/PWM/数值/触发/ADC/定时器/EEPROM）→ 对应代码区域 |
| 无截图退化 | 基于 ⚠️/❌ 功能项定位，或全模块逐项排查 |

#### Step 3D：5维度区域深度排查

| 维度 | 检查内容 | 检查点数 |
|------|----------|----------|
| **数据流追踪** | 变量定义→赋值→使用完整路径，类型/溢出/精度/残留 | 6 |
| **状态机验证** | 状态枚举 + 转移矩阵，不可达/死锁/非法/互斥 | 5 |
| **边界条件** | 循环/数组/除零/空值/极端值/缓冲区/计数器 | 7 |
| **时序逻辑** | 中断优先级/volatile/原子操作/定时器/消抖/DMA | 8 |
| **模块交互** | 全局变量一致/参数范围/头文件匹配/回调注册/初始化依赖 | 7 |

排查后额外执行**修复连带验证**：逐条审查已有修复是否引入新问题 + 交叉影响检查。

#### 经验反哺增强

深度模式发现的新模式额外写入：

```json
{
  "id": "EXP-041",
  "discovery_method": "deep_analysis",
  "analysis_dimension": "数据流追踪",
  "...": "..."
}
```

形成"深度模式喂养经验模式"的正循环 — 深度排查发现的新 bug 类型，下次在经验模式中就能直接命中。

## 核心清单（P0 致命级）

无论经验库如何筛选，以下项目**每次都会检查**：

**嵌入式（6项）**：
1. switch-case 缺少 break
2. 变量类型溢出（uint8_t 存 >255）
3. 整数除法截断（`(float)(a/b)` → `(float)a/(float)b`）
4. sprintf 缓冲区溢出
5. 函数已定义但未注册到调度器
6. sscanf 格式符与目标类型不匹配

**算法（4项）**：
1. 数组越界
2. 整数溢出（int 应用 long long）
3. 格式符与变量类型不匹配
4. 边界条件遗漏（n=0, n=1）

## 报告格式

### 经验模式报告

新建 `fix-report.md`，包含：
- 题目信息（功能要求、外设、约束）
- 截图分类（题目截图 / 错误截图）
- **经验命中统计**（经验库匹配数 / 核心清单命中数 / 新发现数）
- 代码修复详情（原始代码 → 修复代码 + 引用题目要求）
- 功能覆盖核查（✅ / ⚠️ / ❌）
- 原始代码备份

### 深度模式报告（v3.0 新增）

在已有报告末尾**追加** `## 深度排查（第2轮）` 章节：
- 排查策略说明（嫌疑区域数 + 来源）
- 嫌疑区域 × 5维度排查结果表
- 修复连带验证结果
- 新增修复（编号延续，如 修复7、修复8...）
- 功能覆盖核查对比表（第1轮状态 vs 第2轮状态）
- 累计修复汇总（含轮次标记）
